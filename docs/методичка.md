# Webster: методичка проекта (CBL)

Этот документ приводит проект Webster к понятной и рабочей структуре по методологии Challenge-Based Learning (CBL).

## 1. Engage (Вовлечение)

### Big Idea
Дизайн.

### Проблема
Визуальный контент нужен почти в каждом проекте и бизнесе, но создание качественного дизайна требует опыта, времени и инструментов.

### Challenge
Создать веб-редактор, который позволит пользователям без дизайнерского бэкграунда быстро делать качественные макеты и графику.

### Целевая ценность
- Снизить порог входа в дизайн.
- Ускорить создание контента для соцсетей, обучения и малого бизнеса.
- Дать понятный интерфейс и готовые шаблоны вместо сложных профессиональных программ.

### Артефакты этапа
- Краткое описание проблемы и цели проекта.
- Портрет целевой аудитории.
- Список ключевых пользовательских задач.

## 2. Investigate (Исследование)

### Что исследуем
- Конкурентов: Canva, Stencil, Piktochart и другие онлайн-редакторы.
- Предметную область: цветовые палитры, типографика, базовая композиция.
- Форматы: PNG/JPG (растр), SVG (вектор), PDF.
- Технические вопросы: Canvas, drag-and-drop, undo/redo, асинхронные операции.
- Архитектуру: трехуровневая модель (UI, бизнес-логика, данные).

### Организация работы
- Распределить роли по направлениям: продукт, дизайн, frontend, backend, DevOps, QA.
- Сформировать backlog с приоритетами (MVP -> v1 -> v2).
- Определить инструменты планирования и отчетности.

### Артефакты этапа
- Отчет по конкурентам.
- Технические решения по редактору и хранению данных.
- План спринтов и ответственности.

## 3. Act (Действие и разработка)

### Разрешенные технологии по условию проекта
HTML, CSS, TypeScript, React (frontend), NestJS (backend), база данных на выбор, Swagger (OpenAPI), Docker.

### Требования к разработке
- Адаптивность: корректная работа на desktop, tablet, mobile.
- Стабильность: обработка ошибок и отсутствие критических падений.
- Валидация: обязательная проверка входных данных на frontend и backend.
- Код-стайл: единые правила для TypeScript, React, NestJS.

### Архитектура
Рекомендуемая архитектура: три уровня.
- UI-уровень: страницы, компоненты, состояния интерфейса.
- Бизнес-уровень: сценарии работы редактора, правила авторизации, экспорт, шаблоны.
- Data-уровень: база данных, файловое хранилище, кеш/очереди (при необходимости).

### UX/UI процесс (10 шагов)
1. User Persona.
2. Frustrations.
3. Competitor analysis.
4. Strategy.
5. Features and prioritization.
6. User flow.
7. Wireframes.
8. Moodboards.
9. UI design.
10. User testing.

### Обязательный функционал редактора
- Готовые шаблоны и создание проекта с нуля.
- Добавление текста и настройка параметров (цвет, размер и др.).
- Рисование (например, карандаш).
- Фигуры (прямоугольник, треугольник, стрелки и т.д.).
- Добавление изображений на полотно.
- Перемещение элементов мышью и клавиатурой.
- Удаление элементов.
- История изменений (undo/redo).
- Изменение размера canvas и zoom.
- Экспорт в JPG, PNG, PDF и дополнительный формат.
- Шаринг в социальные сети/мессенджеры.
- Пользовательские шаблоны.

### Deploy
- Контейнеризация frontend и backend через Docker.
- Оркестрация через docker-compose.
- Публикация на хостинге с доменным именем.

## 4. Стек (Макс) — адаптированный под Webster

Ниже стек на базе прошлого проекта, но без лишнего для текущих задач.

### Backend
- NestJS + TypeScript.
- MongoDB.
- JWT (access/refresh) для авторизации.
- Nodemailer для подтверждения почты.
- Экспорт"
- Swagger (OpenAPI) для документации API.

### Frontend
- React + TypeScript + Vite.
- React Router.
- Tailwind CSS.
- Axios.
- Zustand.
-  для canvas-редактора.

### DevOps
- Docker + Docker Compose.
- Nginx как reverse proxy и раздача статики.
- Конфигурация окружений через .env.

### Что убрано как лишнее на этом этапе
- Stripe (платежи не входят в текущий scope).
- Google OAuth (можно добавить позже, в v2).
- Nuqs (необязателен для MVP).
- Lexical Rich Text Editor (избыточен для первой версии canvas-редактора).
